# VA LangGraph Production Parity

**Source:** [`src/multi_agents/va_langgraph`](../../src/multi_agents/va_langgraph) compared with production [`va-agents`](../../../va-agents/src/agents).

This checklist tracks what `va_langgraph` must prove before it can be treated as a production-hardened accounting-agent orchestrator. The parent agent owns routing, state, response shaping, telemetry, and eval contracts. Concrete capabilities remain delegated: support help goes through `hc_lg` and `hc_rag`; forecasting runs as a subgraph; MCP accounting tools provide accounting data/actions.

## Architecture Contract

- `va_langgraph` is the parent accounting agent, not a direct replacement for MCP.
- MCP is one tool/capability path under the accounting agent, alongside forecasting, support delegation, direct answers, escalation, and future tools.
- Support help is delegated to `hc_lg`; `hc_lg` owns CRAG, retrieval quality gates, support grounding, and `hc_rag` or Bedrock backend selection.
- Non-support accounting behavior should be tested against deterministic tool results, not URL-grounding datasets.
- Production `va-agents` is the parity reference for user-facing accounting behavior and tool contracts, not a requirement to duplicate support RAG internals inside `va_langgraph`.

## Support RAG Parity

| Area | Production `va-agents` | `va_langgraph` target | Status |
|---|---|---|---|
| Retrieval surface | `fetch_support_knowledge` over Bedrock HYBRID | Delegate to `hc_lg`; `hc_lg` calls Bedrock or `hc_rag` | In progress |
| Input shape | `queries: string[]` generated by the model | User query passed to `hc_lg`; query expansion owned by `hc_lg` | Intentional difference |
| Passage shape | `id`, `score`, `title`, `url`, `text` | Preserve `hc_lg.passages`, `retrieved_urls`, and `_trace` at top level | Implemented |
| Grounding | Passage ID and citation checks in TS callbacks | Shared support-agent grounding in `hc_lg` | Delegated |
| Eval metadata | Eval context maps in invocation context | `PROMPT_VERSION`, `_trace`, `retrieved_urls`, `passages` visible in VA response | Implemented |
| Failure behavior | KB errors escalate/contact support | Delegate failure returns structured `contact_support` fallback | Implemented |

## Production-Hardening Gates

- Boot: `build_graph()` compiles and `/chat` uses async graph invocation for async nodes.
- Contract: top-level VA response includes `answer`, `message`, `sources`, `suggestions`, `contact_support`, `failure_reason`, `PROMPT_VERSION`, `retrieved_urls`, `passages`, and `_trace`.
- Support smoke: `va_lg` can call `hc_lg` locally and preserve delegated support metadata.
- Eval smoke: `agent_caller --mode sa --agent va_lg` completes on the support grounding dataset and extracts the same support metadata shape as direct `hc_lg`.
- Routing: accounting router can select support, forecasting, MCP domains, direct response, escalation, and memory.
- MCP tests: CI uses a deterministic MCP adapter or fixture backend before real accounting tool behavior is considered covered.
- Tool-result grounding: non-support claims are checked against returned tool records, not support article URLs.
- Write actions: create/edit/send/invite paths remain disabled or mocked until a safe test account and confirmation flow exist.

## Synthetic Eval Scope

Synthetic eval rows should cover these dimensions:

- Routing accuracy: expected capability path and domain.
- Tool selection: required MCP/forecasting tool names.
- Field fidelity: response values match fixture records for money, dates, IDs, customer names, VAT, and statuses.
- UI contract: `tableType`, `form`, `confirm`, `navButtons`, and `contactSupport` when those fields are supported.
- Safety: write/send/invite requests require a form, confirmation, or safe escalation.
- Tool-result grounding: every factual claim about accounting records is supported by a fixture result.

Initial seed rows live in [`data/datasets/va_langgraph/eval_sets/accounting_synthetic_seed.jsonl`](../../data/datasets/va_langgraph/eval_sets/accounting_synthetic_seed.jsonl), with fixture records in [`data/datasets/va_langgraph/fixtures/accounting_seed.jsonl`](../../data/datasets/va_langgraph/fixtures/accounting_seed.jsonl).
